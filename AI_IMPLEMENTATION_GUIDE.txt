PART 2 — THE ARCHITECTURE CHANGE: FROM TOOL CALLING TO INTENT DETECTION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

OLD ARCHITECTURE (OpenAI Tool Calling):
  User → GPT decides which tool → We call DB → Results to GPT → Answer
  Cost: 2 API calls per query (tool selection + response generation)

NEW ARCHITECTURE (Intent Detection + Data Injection):
  User → We detect intent (keyword regex) → We fetch DB data → 
  Data + question sent to Groq → Groq formats answer
  Cost: 1 API call per query

THE FLOW IN DETAIL:

  Step 1: User asks "Show transfers above ₹5000 this week"

  Step 2: detectIntent() → returns 'SEARCH'
          extractFilters() → returns { minAmount: 5000, days: 7 }
          (This is pure JavaScript — no AI cost)

  Step 3: We run Prisma query with those filters:
          db.getRecentTransactions(userId, 20, { minAmount: 5000, days: 7 })
          Real data returned from PostgreSQL.

  Step 4: We build a structured prompt:
          "DATA (from database — do not invent any other numbers):
           MATCHING TRANSACTIONS: [real JSON data]
           USER QUESTION: Show transfers above ₹5000 this week"

  Step 5: Groq formats a clean response using ONLY the injected data.
          It cannot hallucinate because the data is right there.

WHY THIS IS BETTER:
  - One API call instead of two (50% cheaper)
  - Intent detection is instant (no network latency)
  - AI is used only for what it's good at: formatting language
  - Database logic stays in Node.js where it belongs
  - Easier to debug (you can see exactly what data was fetched)

PART 5 — HOW INTENT DETECTION WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  INTENT          TRIGGER KEYWORDS              DATA FETCHED
  ─────────────── ───────────────────────────── ─────────────────────────────────
  BALANCE         "balance", "how much have I"  All accounts + total
  RECENT_TXN      "last N transactions"         Recent N transactions
  SPENDING        "spent", "expense", "month"   Spending by period + comparison
  TOP_RECIPIENTS  "who do I", "most transfers"  Top 5 recipients by count
  SCHEDULED       "scheduled"                   All scheduled transfers
  STANDING        "standing", "recurring"       All standing instructions
  BENEFICIARIES   "beneficiary", "payee"        Saved beneficiaries
  SEARCH          "above ₹X", "below ₹X"       Filtered transaction search
  FRAUD           "fraud", "suspicious"         7-day fraud signal analysis
  ADVISOR         "afford", "save", "advice"    Full financial overview
  UPI             "upi"                         UPI ID data
  GENERAL         (fallback)                    Balance + last 5 transactions

  FILTER EXTRACTION (from natural language):
    "above ₹5000"        → { minAmount: 5000 }
    "below ₹10,000"      → { maxAmount: 10000 }
    "last 7 days"        → { days: 7 }
    "last 3 weeks"       → { days: 21 }
    "last 5 transactions" → { limit: 5 }

  All LLM calls go through the llm() wrapper function:

    const llm = async (messages, options = {}) => {
      if (!groq) return null;                    // graceful fallback
      const res = await groq.chat.completions.create({
        model:       MODEL,
        messages,
        temperature: options.temperature ?? 0.3,
        max_tokens:  options.max_tokens  ?? 1024,
      });
      return res.choices[0].message.content.trim();
    };

PART 8 — INTERVIEW QUESTIONS & MODEL ANSWERS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Q: "Why did you choose Groq over OpenAI?"
A: Free tier for development, 3-5x faster inference due to custom hardware (LPUs),
   and Llama 3.3-70B is open-source so there's no vendor dependency.
   The API is OpenAI-compatible so switching takes less than 30 lines.

Q: "Why did you change from Tool Calling to Intent Detection?"
A: Tool Calling requires two API round-trips (one to select tools, one to
   generate the response). Intent Detection is instant JavaScript regex — we
   only need one API call for the final formatting. This cuts latency in half
   and reduces costs by ~50%. For a banking application with structured data,
   intent detection is more predictable than asking the model to pick tools.

Q: "How do you prevent the AI from making up financial data?"
A: The AI receives a system prompt that explicitly forbids inventing numbers.
   More importantly, we inject the real database data directly into the prompt
   and instruct Groq to only answer using what's in the DATA section.
   The database is the single source of truth; Groq is just a formatter.

Q: "What happens if Groq is down or the API key is missing?"
A: The groq client is null when no key is set. Every function that calls llm()
   checks for null and returns static/database-only responses instead.
   Module 3 (Fraud Detection) doesn't use the LLM at all — it's pure Prisma.
   Module 2 (Insights) falls back to static insight cards.
   The app never crashes due to AI unavailability.

Q: "What model are you using and why?"
A: llama-3.3-70b-versatile. It's Groq's recommended general-purpose model,
   supports instruction following, handles Indian number formatting correctly,
   and processes banking queries accurately. The "versatile" variant is
   optimized for multi-turn conversation and structured outputs.

Q: "How does conversation history work?"
A: The last 6 messages from the frontend are included in every API call.
   This allows follow-up questions like "What about last month?" after asking
   about this month's spending. We cap at 6 exchanges to control token costs.

Q: "What is the rate limiter for AI endpoints?"
A: A dedicated aiLimiter of 30 requests per minute per IP is applied to
   /api/ai/* in app.js. This matches Groq's free tier limit and also
   prevents any runaway API cost from a bug or abuse.

Q: "How would you scale this in production?"
A: (1) Cache common queries (balance, recent txns) in Redis with 30-second TTL.
   (2) Queue non-real-time requests (insights generation) via a job queue.
   (3) Add streaming responses for the chat endpoint using Groq's stream API.
   (4) Move to a paid Groq tier or switch to a self-hosted Llama deployment.

PART 10 — HOW TO BUILD THIS FROM SCRATCH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  1. npm install groq-sdk

  2. Create client:
       const Groq  = require('groq-sdk');
       const groq  = new Groq({ apiKey: process.env.GROQ_API_KEY });
       const MODEL = process.env.GROQ_MODEL || 'llama-3.3-70b-versatile';

  3. Write intent detector (keyword-based, no AI cost):
       const detectIntent = (msg) => {
         if (/balance/.test(msg))       return 'BALANCE';
         if (/transaction/.test(msg))   return 'TRANSACTIONS';
         if (/spend|expense/.test(msg)) return 'SPENDING';
         return 'GENERAL';
       };

  4. Write DB fetchers:
       const getBalance = (userId) => prisma.account.findMany({ where: { userId } });

  5. Build the prompt with real data:
       const prompt = `DATA:\n${JSON.stringify(dbData)}\n\nQUESTION: ${userMessage}`;

  6. Call Groq:
       const res = await groq.chat.completions.create({
         model: MODEL,
         messages: [
           { role: 'system', content: 'Only answer using the DATA provided.' },
           { role: 'user',   content: prompt }
         ]
       });
       return res.choices[0].message.content;

  7. Return to frontend — same JSON structure as before.

  That's the complete pattern. The entire migration of a full banking AI
  service from OpenAI tool-calling to Groq intent-detection took ~200 lines.
